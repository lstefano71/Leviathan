# Product Requirements Document (PRD)

**Project Name:**[CodeName: Leviathan] Large-File Text & Hex Editor  
**Document Version:** 1.0  
**Target Platform:** Windows, macOS, Linux (Desktop)  
**Tech Stack:** C# / .NET 10+ (Native AOT)  

## 1. Executive Summary

The goal of this project is to develop a lightweight, ultra-performant, and self-contained desktop application capable of opening, viewing, and editing massive files (50GB+). Unlike traditional text editors that load data into RAM, this application will utilize extreme virtualization, memory-mapped files, and immediate-mode UI rendering. It is designed for developers, data scientists, and security engineers who need to analyze massive log files, database dumps, or binaries without application crashes or latency.

## 2. Target Audience & Use Cases

* **Security Analysts/Reverse Engineers:** Need to inspect and patch massive binary files via Hex view.
* **DevOps/SysAdmins:** Need to open 50GB server log files instantly to search for specific error patterns.
* **Data Engineers:** Need to inspect raw CSV or JSON line dumps before processing.

## 3. Product Requirements

### 3.1. Functional Requirements

* **Dual View Modes:**
  * **Text Mode:** Support for ANSI and UTF-8 encodings.
  * **Hex Mode:** Standard hex editor layout (Offset, Hex grid, ASCII representation).
* **Instant Open:** A 50GB file must open and display the first page of content in under 500 milliseconds.
* **Zero-Latency Editing:** Users must be able to insert or delete characters anywhere in the file instantly.
* **Infinite Scrolling:** Smooth, 60+ FPS scrolling across the entire file size.
* **Dynamic Line Wrapping:** Support visual line wrapping in Text Mode without pre-processing the entire file.
* **Search & Find:** High-speed pattern matching for both raw hex bytes and text strings.
* **Atomic Save:** Saving edits to massive files must not corrupt the original file on failure.

### 3.2. Non-Functional Requirements (NFRs)

* **Deployment:** Single, self-contained executable. No runtime installation required.
* **Footprint:** Binary size under 40MB (Native AOT trimmed).
* **Memory Usage:** Base application RAM must remain under 50MB (excluding OS-managed memory-mapped file caches).
* **Garbage Collection:** Zero allocations (`0 Bytes`) in the core render loop to prevent micro-stutters.

## 4. Technical Architecture & Implementation Specs

To support 50GB+ files, traditional UI and data concepts (like `string`, `File.ReadAllText()`, or standard UI `TextBox` controls) are strictly prohibited. The architecture relies on the following pillars:

### 4.1. Storage & Data Backend: The Piece Table

The file backend will act as a pure read-only data source to prevent massive disk shifts on byte insertion.

* **Original File:** Mapped into memory using `System.IO.MemoryMappedFiles`. The OS handles paging chunks into RAM.
* **Append Buffer:** A RAM-based arena allocator (`ArrayPool<byte>`) storing newly typed characters.
* **Piece Tree:** A self-balancing tree (e.g., Red-Black Tree) tracks the logical document sequence.
  * Every node (Piece) is a `readonly struct` containing: `[Source: Original|Append, ByteOffset, Length]`.
  * Insertions split pieces O(log N) time, ensuring zero-latency typing.
* **Save Mechanism:** Edits stream the pieces in order to a temporary `.tmp` file, followed by an atomic OS-level file swap.

### 4.2. UI Framework & Rendering Loop

The UI layer must bypass retained-mode visual trees.

* **Framework Options:**
  * *Primary Recommendation:* **`Hexa.NET.ImGui`** (Immediate Mode GUI) with `Veldrid` or `Silk.NET`. Best for pure performance, zero-allocation rendering, and strict monospace layout.
  * *Alternative:* **Avalonia UI** for the application shell (menus, dialogs), with a completely custom rendering surface overriding `Render(DrawingContext)` and drawing raw text/shapes via **SkiaSharp** / HarfBuzz.
* **Zero-Allocation Pipeline:** The render loop must extract `ReadOnlySpan<byte>` from the Piece Table and draw directly to the screen. No reference types (`class`) or `string` allocations are permitted per frame.

### 4.3. Text Decoding (ANSI & UTF-8)

Because the viewport targets random byte offsets, the engine must handle mid-character decoding.

* When fetching a chunk of bytes to render, the engine must scan backwards up to 3 bytes to detect valid UTF-8 sequence boundaries.
* Utilize .NET 10 `System.Text.Utf8` and `System.Text.Rune` to decode `ReadOnlySpan<byte>` into screen graphemes natively on the stack.

### 4.4. The Line Wrapping & Navigation Engine

Accurate scrollbars and line wrapping for 50GB files require a hybrid approach:

* **Background Hard-Break Indexing:** Upon opening a file, spin up a background task utilizing .NET 10 Hardware Intrinsics (`Vector256.IndexOf` / AVX2/AVX-512) to scan the memory-mapped file at gigabytes-per-second, locating `\n` (0x0A) bytes.
  * Store every 1,000th line break in a sparse array.
  * This provides O(1) mathematical lookups for the scrollbar thumb position.
* **Just-In-Time (JIT) Visual Wrapping:** Line wrap is calculated *only* for the currently visible viewport.
  * The engine pulls a ~10KB window of bytes around the user's current scroll offset.
  * It calculates column/width thresholds on the fly and caches the visual break points for the current frame.

## 5. Development Phases & Milestones

### Phase 1: Core Engine (Headless)

* Implement `System.IO.MemoryMappedFiles` wrapper.
* Build the `PieceTable` / `PieceTree` structures with `ref struct` and `ReadOnlySpan` support.
* Write unit tests simulating 50GB edits, verifying O(1) performance.
* Implement background SIMD newline indexing.

### Phase 2: Render Pipeline & Hex View

* Set up .NET 10 Native AOT project.
* Integrate UI Framework (`Hexa.NET.ImGui` or Avalonia/Skia).
* Build the Hex View first (simpler than Text mode because formatting is purely mathematical: 16 bytes per line).
* Ensure 60fps scrolling and verify zero GC allocations in the profiler.

### Phase 3: Text View & UTF-8 Handling

* Implement backwards-scanning for UTF-8 boundary detection.
* Implement `System.Text.Rune` iteration rendering.
* Build JIT line-wrapping logic for the visible viewport.

### Phase 4: Polish & System Integration

* Implement keyboard input handling, cursor positioning, and selection highlighting (crossing Piece Table boundaries).
* Add File -> Save logic (streaming pieces to disk).
* Final Native AOT trimming configuration (ILC).

## 6. Success Criteria

* Application binary compiles to < 40MB standalone.
* A 50GB file opens and is visually navigable in < 0.5s.
* Inserting text into the center of a 50GB file registers in < 1ms.
* Memory profiler shows no allocation spikes during continuous scrolling.
