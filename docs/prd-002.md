## Plan: CodeName Leviathan - Large-File Editor

Building a high-performance, zero-allocation large file editor in C# / .NET 10+ using a Piece Table, Memory-Mapped Files, and an immediate-mode GUI (e.g., Hexa.NET.ImGui) for 50GB+ file support.

**Steps**
1. **Initialize Project:** Create the .NET 10 Native AOT core and UI projects.
2. **Phase 1: Core Engine:** Implement `MemoryMappedFile` wrapper, `PieceTable` structure (with `ref struct` / `ReadOnlySpan`), and background SIMD newline indexing.
3. **Phase 2: Render Pipeline & Hex View:** Integrate UI framework (`Hexa.NET.ImGui`), build the Hex View displaying memory-mapped bytes with zero GC allocations per frame. *depends on step 2*
The Hex View will be the first visible output, allowing us to verify the core engine's performance and correctness before moving to the more complex Text View. The Hex View should be split as usual in the typical hex editor style: offset column, hex bytes column, and ASCII column. This will allow us to validate the memory-mapped file handling and Piece Table edits visually while ensuring that we maintain zero allocations during rendering.
4. **Phase 3: Text View & UTF-8 Handling:** Implement JIT line-wrapping, and UTF-8 boundary detection using `System.Text.Rune`. *depends on step 2 / parallel with step 3*
5. **Phase 4: Polish & System Integration:** Implement keyboard input, cursor tracking across Piece Table boundaries, and atomic saving (streaming to .tmp then swap).
Search and replace functionality will be added in this phase, leveraging the efficient indexing from the core engine to perform fast searches even in large files. We will also implement a simple "Find" dialog in the UI to allow users to input search queries and navigate through results. Search in the hex view should be capable of finding byte patters as well as text patterns, while the text view will focus on UTF-8 string searches.

**Verification**
1. Compile the app with Native AOT and verify the output binary size is under 40MB.
2. Generate a 50GB dummy file; measure open time to ensure `< 0.5s`.
3. Use a .NET memory profiler to confirm `0 Bytes` allocated in the continuous render loop while scrolling.

**Decisions**
- The project is greenfield (the git repository is empty).
- Standard UI controls and `string` types are bypassed in favor of raw byte spans and immediate/custom rendering.
- Editing utilizes an Append Buffer + Piece Table instead of shifting bytes on disk.
